Estaba contento con docpad, pero el flujo de trabajo era un poco tedioso: visualizar los cambios toma más de 5 minutos.

Buscando en la red encontré que Hugo presume de ser el más rápido.

Migrar de docpad a Hugo ↑

Los artículos docpad son compatibles con Hugo, pero se necesita entrecomillar las fechas…

# en todas las carpetas que contengan artículos
sed -i -e 's/date: \([0-9-]*\)$/date: "\1"/' *.md

…y renombrar los archivos:

# desde la carpeta superior
for f in $(find -name '*.html.md'); do mv "$f" "${f%.html.md}.md"; done

Generar un nuevo sitio ↑

Hugo genera automáticamente los archivos necesarios para empezar con el comando:

hugo new site nombre-del-sitio

Antes de empezar ↑

Pero antes de empezar, hay que generar el estilo de las páginas:

  • layouts/_default/single.html: genera las páginas individuales, y para que esto funcione bien debe ser una página válida que contiene la línea:

      {{ .Content }}

  • layouts/_default/list.html: genera listados de páginas individuales, también debe ser una página válida, que haga referencia a los artículos de alguna forma, por ejemplo:

      {{ range .Data.Pages }}<a href="{{ .Permalink }}">{{ .Title }}</a> {{ end }}

Temas ↑

Me costó un poco entender cómo usar los temas. Aunque hay varios temas bonitos en el sitio oficial, la mayoría requiere varios pasos previos para funcionar, que son específicos para cada tema.

Los temas se hacen agregando páginas válidas en la carpeta themes/nombre-del-tema/layouts que se genera con el comando…

hugo new theme nombre-del-tema

…con los elementos que necesita dentro de themes/nombre-de-tema/static.

En mi opinión, los temas deberían tener lo necesario, para empezar a usarlos sin problemas.

Vayamos por partes ↑

Una ventaja de Hugo es que permite reutilizar partes del contenido web, si se guardan los pedacitos en el directorio layouts/partials/ y se utiliza en otro plantilla la función:

{{ partial "nombre-de-plantilla.html" . }}

El punto del final es importante porque con él se envían las variables de la página.

Pedazos de código comunes ↑

Puedes definir pedazos pequeños de código para reutilizar en tu sitio que se llaman códigocortos (shortcodes).

{{% nombre parámetro1 … %}} # procesa el contenido con markdown
{{< nombre parámetro1 … >}} # no procesa el contenido
  • Los códigocortos pueden anidarse (doc, ejemplo)

Conviene agregar gestión de errores en los códigocortos.

Opciones de procesamiento de markdown ↑

Se puede configurar HUGO para usar plantillas para algunos elementos de markdown:

  • enlaces,
  • imágenes,
  • encabezados

Poderes mágicos ↑

Aunque dejar de usar un sistema de gestión de contenidos dinámicos (SGCD) puede sonar como un retroceso, con los SCGE se puede hacer casi todo lo que con los SCGD pero sin necesidad de una base de datos.

Si se necesita en algún momento un sistema de alta transaccionalidad, se enlaza un servicio aparte.

Se pueden usar las funciones de Hugo para hacer las cosas complejas que se quieran agregar a la página.

Y se pueden conectar servicios extra para las funciones interactivas.

Referencias cruzadas ↑

Para referenciar un artículo hecho en Hugo o partes de él, se utiliza la función:

{{< relref "/ruta/al/artículo.md#sección" >}}

La sección se genera automáticamente de los encabezados, o se puede indicar directamente en markdown así:

# Encabezado{#referencia}

Redirigir desde HUGO ↑

Los sitios de SEO recomiendan usar redirecciones al mover sitios web, por eso investigué acerca de cómo redireccionar páginas en HUGO y me encontré este plugin que lo resuelve así como el post que explica cómo funciona:

# file: /ruta/a/redirección.md
---
type: redirect
url: /url/a/redirigir
redirect_to: /url/destino
redirect_enabled: true
---

En principio, agregar una redirección por cada página que mueves, mantiene tu puntaje de SEO.

Yo tengo una plantilla que uso con new redir.

Migración desde HUGO a pandoc ↑

HUGO usa TOML como formato para metadatos, migrar hacia pandoc resulta ser muy sencillo, solo es necesario cambiar TOML por YAML:

find -L content \
	-type f \
	-name '*.md' \
| xargs sed -i -r \
	-e 's#^\+\+\+[ \t]*$#---#' \
	-e 's#([^ =]+) = #\1: #'

Embeber archivos PDF ↑

Utilizo un shortcode para embeber archivos PDF:

{{< pdf "/ruta/al/archivo.pdf" >}}

Es importante que se use {{< >}} en vez de {{% %}} para evitar que se elimine el código HTML y obtengamos <!-- raw HTML omitted --> en la página. La solución recomendada en StackOverflow es insegura.

Si se requiere markdownificar, usar un shortcode para hacerlo explícito en vez de desactivar la seguridad de Hugo globalmente.

El shortcode pdf.html:

<embed src="{{ .Get 0 }}#opciones" attibutos>

Otras formas para usar PDF en HTML:

  • <etiquetas> HTML y sus variantes, se usan principalmente object, iframe embed.

    Los navegadores suelen ignorar los atributos de apertura de PDF.

  • Código JS.

Incluir archivos ↑

Se pueden generar páginas formadas de varios archivos usando como archivo principal en el directorio index.md en vez de _index.md.

¿Cómo se usan luego los archivos .md en un paquete de hoja (leaf bundle)?

El autor de Hugo recomienda shortcodes o mmark pero mmark ya no se usa.

  • repo mmark.
  • ejemplos de shortcode.
  • inline de svg.
  • mi implementación.

Para usar mi shortcode uso:

{{% include "/path/to/filename/from/content" %}}

Precauciones al incluir archivos ↑

  • Si la ruta al archivo contiene acentos o caracteres fuera de ASCII, no funciona.
  • Al principio requería que el archivo tenga extensión .md pero entonces el sitio fallaba porque intentaba generar páginas a partir de esos archivos (incluso en un directorio con un index.md):
    • Si no había metadatos, el sitio daba error al generar la página.
    • Si tenía metadatos, al incluirse salían.

Se puede agendar la publicación ↑

Se puede usar publishDate para elegir la fecha de publicación y expiryDate para que deje de publicarse la página.

Para que eso sea posible, debe automatizarse la actualización diaria del sitio. Se puede automatizar la publicación en un punto del tiempo usando cron-job (y monitorear).
