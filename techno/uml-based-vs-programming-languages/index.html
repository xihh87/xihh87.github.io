Model Driven Architecture no evolucionó, los lenguajes sí.

Escuchando la clase de Yegor acerca de UML, me da la impresión de que siempre tuvimos el mismo sueño:

Y a pesar de que programar se ha vuelto mucho más sencillo porque ahora no tenemos que entender:

  • el funcionamiento interno del hardware (binario),
  • las instrucciones específicas del microprocesador (ensamblador),

o cuidar:

  • el flujo de ejecución (GOTO)
  • el manejo de la memoria (C)
  • la gestión de los hilos (Go, Rust)

Hay una diferencia entre el diseñar todo el sistema esperando que funcione, y resolver el problema inmediato, iterando poco a poco.

No funcionó automatizar el código por medio de diagramas, posiblemente porque no evolucionó orgánicamente.

La idea de que podemos dibujar diagramas y se generarán programas no parece haber evolucionado resolviendo problemas poco a poco, mejorando poco a poco el dominio del servicio.

Ahora tenemos más facilidades para escribir código:

  • Lenguajes mucho más poderosos y simples,
  • Muchos tutoriales y ejemplos en internet

y no necesitamos entender cómo funcionan las cosas hasta que algo no funciona como esperamos:

En ese momento, necesitamos entender cómo funcionan los niveles inferiores para poder resolver problemas.

Ahora está de moda intentar que las máquinas hagan el código y se están obteniendo resultados impresionantes.
