Una conferencia acerca de cómo hacer microservicios ↑

Es necesario usar virtualización y elasticidad para gestionar microservicios efectivamente.

El diseño es modular y encapsula los datos.

Una analogía de microservicio es un organo que funciona como parte de un sistema.

Qué forma un microservicio: ↑

Los servicios necesitan tomar datos de una base de datos, se requiere un servicio, se requiere un caché para mejorar la eficiencia.

Los problemas vienen de hábitos:

Dependencias ↑

Si hay un mal servicio, puede generar un problema.

Hystrix: proteger la aplicación, dar un buen fallback.

Probar fallas.

Bibliotecas de clientes: permiten generar código independiente y útil. Pero generan parasitismo. Simplificar al mínimo estas bibliotecas.

Persistencia ↑

Consistency Availability Persistence

escalas ↑

Stateless service. Frequently accesed metadata No instance affinity Loss of a node is a non-event

Replication. Auto scaling. Minimum -> Maximum.

Applying chaos.

Statefull services ↑

Databases & caches. Loss of a node is a notable event.

Do not use dedicated shards. Redundancy is fundamental.

EVCache Writes (spread your caches) read locally, but fallback to read outside.

Excesive Load ↑

Even when cached, try to be efficient. Fail fast.

  • workload partitioning (level for cache).
  • Request level caching.
  • Secure token fallback.

varianza en la arquitectura ↑

Increases the complexity of the environment.

Operational drift (unintentional) ↑

Continuous learning and automation. Analysis -> best practice.

(Use sympatic and parasympatic.)

Polyglot & containers (intentional) ↑

Node management is not a solved problem. Raise awareness of costs. Prioritize by impact. Seek reusable solutions.

Continuous Delivery for Enterprise

Design to automated canaries.

cambio ↑

Automated delivery. Integrated practices.

Unificar en el cliente y refactorizar la organización.

Solutions first, tema second.

Health depends on discipline and chaos.
