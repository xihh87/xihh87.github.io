Los proyectos deben tener un objetivo claramente documentado a partir del cuál se pueden establecer tareas.

Generar tareas ↑

La lista de cosas que tienen que hacerse para lograr el objetivo se registra como tareas en el proyecto.

Las tareas deben tener información suficiente para que cualquier persona capacitada las realice. Cuando agregamos una tarea hay que responder las siguientes preguntas:

  • ¿Qué se necesita hacer?

  • ¿Dónde se va a guardar el resultado?

  • ¿Dónde está la información que se necesita para realizar la tarea?

  • ¿Cuál es el tiempo límite para hacerlo?

  • ¿Por qué es importante?

El líder del proyecto puede asignar tareas a quiénes estén trabajando en un proyecto, o los integrantes del equipo pueden seleccionar una tarea:

Elegir una tarea ↑

Elegir el proyecto o la tarea más relevante.

Verificar si la tarea puede realizarse.

En caso de que haga falta información o contexto, comentar la tarea y referirse a quién la asignó (@persona).

En caso de que haga falta realizar una tarea antes, registrarla mencionando el número de esta tarea.

Crear una nueva rama para trabajar y permitir la revisión.

El nombre de la rama debe iniciar con el número de la tarea, puede ser sólo el número o tener hasta dos palabras para describirla.

Trabajar en la rama de la tarea ↑

Guardar el trabajo cada que haya avance, o antes de hacer experimentos.

Cada avance debe tener el número de tarea, la descripción del trabajo que se hace y si es necesario una explicación más a fondo.

Se puede hacer una rama adicional para hacer experimentos.

Solicitar revisión cuando se haya terminado la tarea (merge request).

Revisar trabajo ↑

La revisión generalmente la hace una persona experimentada. Como mínimo, revisar:

  • El título del commit comienza con el número de tarea (p.e. #5).

  • Cada commit describe el trabajo hecho.

  • Cada commit tiene trabajo independiente.

Textos ↑

  • Buena ortografía.

  • Se entiende lo que se expresa.

  • Es lo más corto que puede ser.

  • El texto está enfocado al público objetivo.

  • No hay espacios en blanco al final de las líneas.

  • Escribir una idea por renglón para facilitar las revisiones.

  • Verificar que GitLab lo muestra correctamente.

Procedimientos ↑

  • La información es suficiente para lograr el objetivo.

  • No hay pasos redundantes.

  • Las imágenes no tienen información extra y son pequeñas.

Código ↑

  • Cada commit resuelve un problema pequeño.

  • El código pasa las pruebas automáticas.

  • No hay espacios en blanco innecesarios.

  • El estilo es consistente con lo demás.

Nombres de archivo ↑

Para leer más fácil y evitar problemas con los nombres, escribir todos los nombres de archivo con minúsculas y guión medio - en vez de espacio.

Los archivos de texto deben terminar con .md para que GitLab los muestre bonitos.

Verificar localmente tu configuración ↑

Se puede usar gitlab-ci-local para verificar el funcionamiento de tu configuración de GitLab.
